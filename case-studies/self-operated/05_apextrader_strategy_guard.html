Case Study 05 · Automated Risk Control

5 Unprofitable Trading Strategies Blocked Before They Fire.

ApexTrader's engine is paused. Its guard isn't. The StrategyGuard circuit-breaker is the proof point — and the pattern it uses applies to any automated system running on real stakes.

Context: ApexTrader (the trading engine) is currently paused pending a guard-restoration check. The StrategyGuard system described here is live and functioning correctly. This report covers the guard architecture — not trading performance, not returns. No P&L claims are made.

The Pain

Automated trading systems have a specific failure mode: they run in the background, doing exactly what they're told, including running losing strategies that a human would have stopped if they'd been watching. The naive fix is to watch constantly. The right fix is to build a circuit-breaker that watches for you — one that knows the difference between a normal drawdown and a strategy that has structurally stopped working, and kills the second kind before it compounds.

The problem isn't unique to trading. Any automated system that takes consequential actions on a schedule — sending emails, placing orders, billing customers, firing webhooks — needs a layer that asks "should this still be running?" before each execution, not just "can this run."

The Build

  • StrategyGuard state machine: each trading strategy tracked in state/strategy_stats.json carries a live risk-control state: ACTIVE, MONITOR, or BLOCKED. State transitions are driven by rule evaluation against rolling performance windows (win rate, expectancy, consecutive losses, daily drawdown). Source: LLC/ApexTrader/.
  • 5-criteria evaluation per strategy: before any strategy is allowed to fire, the guard checks win rate (trailing 20 trades), expectancy (average profit per trade), max consecutive losses, daily drawdown vs. hard limit, and net P&L trend over a rolling window. Failing any single criterion elevates state to MONITOR; failing two or more triggers BLOCKED.
  • Hard block enforcement: BLOCKED strategies cannot execute trades regardless of market signal. The block is only cleared by explicit operator review and sign-off — not by time, not by a subsequent winning trade. Source of truth: state/strategy_stats.json. The file is write-protected from all automated processes; only the operator can change BLOCKED state.
  • Cockpit integration: the Apex tile in the unified cockpit shows live guard state per strategy — GREEN (ACTIVE), YELLOW (MONITOR), RED (BLOCKED). As of April 2026: 5 strategies in BLOCKED state, engine paused, all guard states correctly reflecting known performance issues.

The Takeaway

The guard pattern works for any automated system running on real stakes. The specific domain (trading) is less important than the design principle: separate the "can this run" check (authentication, connectivity) from the "should this run" check (is this still performing as intended?). Most automated systems have the first. Almost none have the second. The second is the one that prevents compounding damage in the background while the operator is focused elsewhere. Build the guard before you need it — after you need it is too late.

Running any automated system that takes consequential actions — outreach sends, billing, order fulfillment? A guard layer is cheap to build and expensive to not have.
